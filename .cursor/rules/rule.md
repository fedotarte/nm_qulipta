## Контекст проекта

Проект: Next.js 16 (App Router) + TypeScript.
Цель: главная страница + страницы статей, которые предопределены в конфиге (источник контента внутри репозитория; внешнего конфига/ CMS пока нет).

## Основные принципы

- Делаем просто, предсказуемо и расширяемо.
- Сначала десктоп, затем мобильная версия.
- Верстка: только CSS Modules (чистый CSS), без UI-библиотек.
- Минимизируем media queries; предпочтение резиновым техникам (clamp/min/max/grid auto-fit).

## Архитектура (Next.js App Router)

- Используем App Router (`src/app`).
- По умолчанию компоненты серверные. `"use client"` добавлять только при необходимости (состояние/эффекты/обработчики событий, браузерные API).
- Общие элементы (Header/Footer) — в `src/app/layout.tsx`, чтобы не дублировать на страницах.
- Статьи: роут `src/app/articles/[slug]/page.tsx`.
  - Страницы статей должны генерироваться из локального источника данных (конфига).
  - Используем `generateStaticParams()` для предопределённых slug.
  - Если slug не найден — `notFound()`.

## Источник контента (конфиг)

- Контент хранится в коде (например, `src/content/*`), как типизированные структуры.
- Должны быть функции:
  - `getAllArticles()`
  - `getArticleBySlug(slug)`
  - `getArticleSlugs()`
- Не подключаем внешние запросы/БД до появления явного требования.

## Лейаут страницы (образец: nm_landings/glau)

- Структура главной страницы приведена к проекту **glau** (соседний репозиторий в `nm_landings/glau`).
- Обёртка: `div.page` (min-height 100vh, flex column, overflow-x hidden), внутри — `main.main` (flex: 1, position relative, z-index 1). Стили в `src/app/page.module.css`.
- Hero-секция: заголовок + фоновое изображение + слот `children` для блока карточек под героем (аналогично glau: Hero + ArticleSlider). Компонент `HeroSection` принимает `children` (например, `HeroCards`).
- Переменная `--z-header` в globals для консистентности с glau. При доработке хедера/мобильного меню можно опираться на glau (drawer, body.drawer-open).

## Верстка и стили (CSS Modules)

- Один компонент = один `*.module.css`.
- Классы в CSS Modules именуем в одном стиле (рекомендуется `camelCase`).
- Запрещено:
  - inline-styles (кроме крайне редких случаев)
  - глобальные стили внутри modules (кроме `:global(...)` при необходимости)
- Layout-подход:
  - desktop-first: базовые стили — для десктопа
  - адаптив через `min()`, `max()`, `auto-fit/minmax`, `flex-wrap`, `gap`
  - media queries допускаются точечно (когда без них нельзя)

## Дизайн-токены / переменные

- Все базовые переменные (цвета, фоны, радиусы, контейнер, отступы) — в `src/app/globals.css` в `:root`.
- В CSS Modules используем только эти переменные для повторяющихся значений.
- Цветовые схемы: не внедряем dark-mode, если это не требование; если нужно — делаем через переменные, без дублирования стилей.

## Компоненты

- Переиспользуемые компоненты — `src/components/*`.
- Компонент:
  - минимальный публичный API (props)
  - стили только через module css
  - без лишней "универсальности" заранее

## TypeScript

- Всё типизируем (особенно контент-конфиг и его функции).
- Не используем `any` без причины.
- Для публичных структур данных — экспортируем `type`/`interface`.

## Доступность и качество

- Семантика: `header/main/footer`, заголовки по иерархии.
- Изображения через `next/image` там, где уместно; у всех картинок есть `alt`.
- Линки — через `next/link` для внутренних переходов.

## Что делает ассистент (Cursor) при задачах

- Перед реализацией уточняет недостающие требования (секции главной, формат статьи, поля конфига).
- Предлагает минимальный набор файлов и изменений.
- Не вводит новые зависимости без согласования.
- Не меняет существующий стиль проекта без причины (нейминг, структура папок, подход к стилям).
